Entity-Relationship (ER) Model

Back to Database Management and Warehousing Topics

The Entity-Relationship (ER) model is a conceptual framework used to describe the structure of a database. It uses entities, attributes, and relationships to represent data and its relationships.

Entities

An entity is a real-world object or concept that can be distinctly identified. Each entity has attributes that describe its properties. Entities are usually represented as rectangles in ER diagrams.

Attributes

Attributes are properties or characteristics of an entity. They are usually represented as ovals connected to their respective entity rectangles in ER diagrams. There are several types of attributes:

  • Simple Attribute: Cannot be divided further (e.g., age).
  • Composite Attribute: Can be divided into smaller sub-parts (e.g., full name can be divided into first name and last name).
  • Derived Attribute: Can be derived from other attributes (e.g., age can be derived from date of birth).
  • Multi-valued Attribute: Can have multiple values (e.g., phone numbers).

Relationships

Relationships are associations between entities. They are represented as diamonds in ER diagrams and connect entity rectangles. There are three types of relationships:

  • One-to-One (1:1): Each entity in the relationship will have exactly one related entity (e.g., a person has one passport).
  • One-to-Many (1:N): One entity is related to multiple entities (e.g., a mother can have many children).
  • Many-to-Many (M:N): Multiple entities are related to multiple entities (e.g., students and courses).

Keys

Keys are attributes or sets of attributes that uniquely identify an entity. There are several types of keys:

  • Primary Key: A unique identifier for an entity.
  • Candidate Key: A set of attributes that uniquely identify an entity and can be chosen as the primary key.
  • Foreign Key: An attribute in one table that refers to the primary key of another table.

Example

Consider a university database with entities such as Student, Course, and Instructor. The relationships could be:

  • Students enroll in courses (many-to-many).
  • Instructors teach courses (one-to-many).

Attributes for Student might include StudentID, Name, and Major. Attributes for Course might include CourseID, Title, and Credits. Attributes for Instructor might include InstructorID, Name, and Department.

Conclusion

The ER model is a powerful tool for designing and representing databases. It provides a clear and organized way to represent entities, attributes, and relationships, making it easier to understand and implement complex databases.
